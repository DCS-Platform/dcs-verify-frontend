Integration guide · R+2 standard

Install DCS Verify in minutes.

Emit an Ed25519-signed receipt the moment something happens, anchor it on Base, and let anyone verify it — without trusting your servers or ours. This is the end-to-end path from API key to an embeddable trust badge.

Verification is free & public ~5 minutes to first receipt
Overview

Five steps, each independently provable.

Signing, RFC 8785 canonicalization, SHA-256 content addressing, hash-chaining, Filecoin storage, and Base anchoring all happen automatically when you emit a receipt. Your job is three API calls and one <script> tag.

01

Get your issuer key

Create an issuer in the issuer console and copy your API key. Your Ed25519 public key is published to an on-chain identity layer (a soul-bound token on Base mainnet), so verifiers confirm your signatures without ever contacting DCS.

# keep the secret key server-side only — never ship it to a browser
export DCS_API_KEY="dcs_sk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxx"
export DCS_AGENT_ID="0348"   # your issuer / agent id
02

Emit a signed receipt

Call the API at the moment an action occurs. The response is a portable receipt: canonicalized (RFC 8785), hashed to a content ID (SHA-256), Ed25519-signed, hash-chained to your previous receipt, Filecoin-stored, and queued for the next Base anchor — all server-side.

curl -X POST https://api.dcslabs.ai/v1/receipts/issue \
  -H "Authorization: Bearer $DCS_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "0348",
    "kind": "agent_action",
    "payload": { "action": "refund_approved", "order_id": "A-1029", "amount": 49.00 }
  }'
// → response
{
  "receipt_id": "r2_a83f12cd",
  "cid":        "sha256:9f3c1b2a…ddee",
  "signed":     true,
  "prev_receipt_cid": null,
  "anchor":     "pending",
  "issued_at":  "2026-06-11T15:42:08.123Z"
}
The exact request schema for your plan — and ready-made snippets in JS, Python and Go — are shown in your issuer console. The receipt shape itself follows the open R+2 standard.
03

Verify it — independently

Anyone can re-check a receipt with nothing but the receipt and your public key. No DCS account, no vendor trust. Fetch the receipt, resolve the issuer key (from the API or directly from the on-chain SBT), and run the open-source reference verifier.

# 1. fetch the receipt
curl https://api.dcslabs.ai/v1/receipts/r2_a83f12cd > receipt.json

# 2. get the issuer's public key (trustless: read it from Base directly)
cast call 0x0A0e824bcA178669C9453c5d72B7AE9120e1985d \
  "agentPubkey(string)" "0348" --rpc-url https://mainnet.base.org

# 3. verify — no install needed
npx dcslabs-r2-verify --receipt receipt.json --pubkey $AGENT_PUBKEY
✓ Schema
✓ Spec version  r2/v0.1
✓ Pubkey match  u4yK_lH8Z6vJ…
✓ Signature

✓ Timestamp     within window

Receipt verified.
04

Embed the Verify badge

Drop a one-line script on any page to show live verification status — an SSL-padlock equivalent for credentials. It renders the current state and links to the public verify page.

<script src="https://verify.dcslabs.ai/badge.js"
        data-cid="sha256:9f3c1b2a…ddee"></script>
05

Manage status over time

Revoke, suspend, reactivate, or renew a credential. Status lives in a privacy-preserving Bitstring list anyone can read — revocation is permanent, suspension reversible. Verifiers see five honest states: valid, expired, suspended, revoked, renewed.

curl -X POST https://api.dcslabs.ai/v1/receipts/r2_a83f12cd/status \
  -H "Authorization: Bearer $DCS_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "status": "revoked", "reason": "superseded" }'
The guarantee

What a verifier trusts — and what they don't.

Every receipt carries its own proof. A skeptic re-runs the math from the raw materials: recompute the CID (RFC 8785 + SHA-256), check the Ed25519 signature against the on-chain key, read the status bit, and confirm the anchor on Basescan. None of that requires DCS to be online or honest.

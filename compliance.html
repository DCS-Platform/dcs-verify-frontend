Verification Test Matrix

DCS supports two on-chain anchoring models. Production uses Merkle batching for efficiency; individual per-receipt anchoring is supported for higher-assurance contexts. This page records our verification tests — every Passed row links to on-chain proof anyone can re-check.

Standard: R-Series (R+2 Ed25519 + RFC 8785 + SHA-256) · Anchor chain: Base mainnet · Last updated: 13 June 2026

Two anchoring models — both tested

Model A · Production

Merkle-batched anchoring

Many receipts fold into one Merkle root; one Base transaction anchors the whole batch. Cheap, fast, scales to millions.

N receipts → N hashes → 1 Merkle root → 1 Base anchor
Model B · High-assurance

Individual per-receipt anchoring

Every receipt gets its own Base transaction — maximum granularity for regulated environments that require per-action on-chain proof.

N receipts → N hashes → N Base anchors
Why two? Production stays on Model A because it is cheaper, faster, and scales far better. Model B is proven so that when a bank, university, government, or enterprise asks “can you individually anchor every action?” the answer is: yes — already tested and verified, available on request.

How to re-verify any result yourself

1 · Open the session link below — it loads the full receipt chain from api.dcsai.ai in your browser.
2 · Click “View all N receipts” to see every receipt hash (CID). Each links to its raw signed JSON on IPFS/Filecoin.
3 · Open the Base anchor on Basescan — the on-chain transaction carries the Merkle root.
4 · Recompute: the receipt hashes fold into the Merkle root; the root matches the anchored transaction. No DCS account, no trust required.
